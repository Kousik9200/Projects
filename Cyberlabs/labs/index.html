LAB-01 // DETECTION ENGINEERING

Detection-as-Code — Sigma Rule Pipeline

Write a production Sigma rule for Mimikatz detection, validate ATT&CK tagging, convert to Splunk SPL and Sentinel KQL, then simulate automated CI/CD deployment to both SIEMs.

$ python lab1_detection_engineering.py
// EXERCISE 1
Write a Sigma Rule
Author a Sigma YAML rule targeting Mimikatz T1003.001 (LSASS credential dump). Include logsource, detection with multiple selections, and proper tags for tactic + technique. The script saves it to lab1_output/mimikatz_detection.yml.
// EXERCISE 2
ATT&CK Tag Validation
Run the validator against your rule. It checks required fields (title, description, detection, level), verifies all attack.tXXXX technique IDs exist, confirms at least one tactic tag, and validates severity level. Fails CI if any check fails.
// EXERCISE 3
Convert to Splunk SPL
The converter reads your Sigma YAML and maps field modifiers (|contains, |endswith) to SPL equivalents (field="*value*"). Output is a fully runnable saved search with index=windows scope, table projection, and sort. Saved to lab1_output/mimikatz_detection.spl.
// EXERCISE 4
Convert to Sentinel KQL
Same rule converted to KQL for Microsoft Sentinel. Field modifiers map to KQL operators (contains, endswith). Output targets DeviceProcessEvents table with where TimeGenerated >= ago(1h) windowing. Saved to lab1_output/mimikatz_detection.kql.
⚡ Challenge: Write Your Own Rule
Edit lab1_output/YOUR_RULE_challenge.yml and write a detection for one of: T1078 (Valid Accounts — impossible travel), T1190 (Exploit Public-Facing App — web shell upload), or T1071.004 (DNS C2 beaconing). Validate it passes all checks, then convert to both SIEM targets.

LAB-02 // CLOUD SECURITY

CIS Benchmark — AWS Misconfiguration Audit

Audit a simulated production AWS account across IAM, S3, security groups, and CloudTrail against 18 CIS AWS Foundations Benchmark v1.5 controls. Score every finding and build a phased remediation plan.

$ python lab2_cloud_security_audit.py
// EXERCISE 1
Inspect the Mock AWS Environment
The mock environment contains a production account with intentional misconfigurations: root MFA disabled, users without MFA, a wildcard IAM policy (Action:* Resource:*), a public S3 bucket, SSH/RDP open security groups, and a single-region CloudTrail without log validation.
// EXERCISE 2
CIS Benchmark Audit
Four audit modules run in sequence: audit_iam() (5 checks), audit_s3() (15 checks across 4 buckets), audit_security_groups() (checks all 5 SGs for dangerous ports and all-traffic rules), and audit_cloudtrail() (2 checks). Each finding maps to a CIS control reference.
// EXERCISE 3
Risk Scoring
Every finding gets a numeric risk score: base_severity × category_weight × env_multiplier. Production accounts multiply by 1.5×. IAM and S3 categories carry a 1.2–1.3× category weight because they directly expose data. A Critical IAM finding in prod scores up to 10.0 × 1.3 × 1.5 = 19.5.
// EXERCISE 4
Remediation Planning
Findings are bucketed into 4 phases: IMMEDIATE (Critical — patch within 24h), THIS WEEK (High), THIS MONTH (Medium), and BACKLOG (Low). Each item includes a specific remediation action with effort estimate. The plan is saved as JSON for Jira import or Notion tracking.
⚡ Challenge: Find the 3 Hidden Findings
Replace MOCK_AWS_ENV with CHALLENGE_ENV in run_lab() and re-run. Three additional findings are embedded: a CloudTrail log bucket with public access, an additional user with a wildcard IAM policy, and a service account with an over-privileged inline policy. Can you find all three?

LAB-07 // ZERO TRUST & NETWORK SECURITY

Zero Trust Architecture — Design, Enforce & Validate

Design a 3-VLAN Zero Trust network lab (Dev / Prod / Management). Build a default-deny policy engine that enforces mTLS and identity for every flow. Simulate 6 attacker lateral movement scenarios and score your architecture against NIST SP 800-207 and the CISA Zero Trust Maturity Model.

$ python lab7_zero_trust_network.py
// EXERCISE 1
Zero Trust Topology Design
Model three network zones: VLAN_10_DEV (trust=low), VLAN_20_PROD (trust=high), and VLAN_30_MGMT (trust=critical). Each zone has a defined subnet and host roster. The key principle: no zone implicitly trusts any other — even prod-web-01 cannot reach prod-db-01 without an explicit policy.
// EXERCISE 2
Default-Deny Policy Engine
Six explicit ALLOW policies are defined. Every other flow hits the default DENY. Each policy enforces: source zone/host, destination, port, whether mTLS cert is required, and which service identity must be presented. The engine evaluates all conditions in sequence and logs every decision for audit.
// EXERCISE 3
mTLS Certificate Design Review
The lab analyzes your PKI design: CA key algorithm (ECDSA P-384), service cert TTL (all ≤24h), identity cert TTL for humans (≤8h), and MFA requirement for human certs. Short-lived certs reduce blast radius if a cert is stolen — the attacker's window is capped at the TTL.
// EXERCISE 4
Lateral Movement Simulation
Six attack scenarios test your policy engine. Notably: an attacker who steals the svc-web cert still gets DENIED when trying to hit the database — because the DB policy requires identity=svc-app, not svc-web. Identity-aware policies stop credential theft from enabling lateral movement.
